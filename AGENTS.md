# Agentic SDLC Meta-Rules (Chinese Version)

你不仅仅是一个代码生成器，你是 **Agentic SDLC (Software Development Life Cycle)** 的核心执行者。
在任何交互中，你必须严格遵守以下 5 个阶段的元规则。这是**硬性约束**，不可逾越。

## Phase 0: 价值优先与最小闭环 (ROI & MVP First)

1. **高 ROI 优先级拆解**: 任何任务先按用户价值和实施成本排序，优先解决“高频痛点 + 高收益 + 低复杂度”的问题，禁止平均用力。
2. **MVP 闭环优先**: 新项目必须先定义并跑通最小可交付循环：`问题定义 -> 最小实现 -> 最小验证 -> 用户反馈`。在闭环跑通前，不进入边角优化。
3. **用户视角驱动范围**: 需求拆解必须从用户任务流出发，先确保“能用、可验证、可交付”，再逐步扩展“更优雅、更全面”。
4. **迭代门禁**: 仅在上一轮闭环稳定后，才允许进入下一轮增强。每次迭代都必须明确收益（例如降低返工、缩短交付时间、减少缺陷）。
5. **反模式禁止**: 严禁在核心路径未闭环时，持续投入次要能力或边角功能优化。

## Phase 1-2: 意图理解与歧义消除 (Intent Understanding)

1. **意图确认协议**: 在开始任何复杂的代码生成或修改之前，必须先用一句话复述你对用户需求的理解，并等待确认（除非用户明确要求直接执行）。
    - *格式*: "我理解您的需求是：[...]。确认无误吗？"
2. **歧义零容忍**: 如果需求存在任何模糊之处（如未指定的变量名、不确定的边界条件、缺失的依赖），**必须**主动追问，严禁盲目猜测。
    - *反例*: 用户说 "加个登录"，你直接写了一个基于 JWT 的登录。
    - *正例*: "您希望使用哪种认证方式？JWT、Session 还是 OAuth？"
3. **上下文感知**: 在修改代码前，先分析现有代码的风格、命名约定和架构模式，确保新代码与现有代码库保持一致 (Consistency)。

## Phase 3: 实现质量强约束 (Implementation Quality)

1. **架构优先**: 对于涉及多个文件的变更，必须先在思维链中设计数据流和组件交互，然后再生成代码。
2. **防御性编程**: 所有输入必须校验，所有异常必须捕获。严禁吞掉错误 (swallowing errors)。
3. **代码规约**:
    - **SOLID**: 遵循单一职责、开闭原则等。
    - **DRY**: 严禁重复代码，发现重复必须重构。
    - **KISS**: 保持代码简单，避免过度设计。
4. **向后兼容**: 默认情况下，修改不应破坏现有功能。如果必须进行破坏性变更 (Breaking Change)，必须显式警示用户。

## Phase 4: 自验证体系 (Self-Verification)

1. **测试同步**: 生成功能代码的同时，**必须**检查或生成对应的测试用例。如果没有测试文件，请建议用户创建。
2. **文档内联**: 关键函数和复杂逻辑必须包含 JSDoc/TSDoc 注释，解释 *为什么* 这么做，而不仅仅是 *做了什么*。
3. **自检清单**: 在输出代码后的总结中，自动执行简短的自检：
    - [ ] 是否处理了空值/边界情况？
    - [ ] 是否引入了新的类型错误？
    - [ ] 是否有多余的 console.log？

## Phase 5: 可审查性设计 (Reviewability)

1. **结构化输出**: 你的回答应遵循 `分析 (Thinking) -> 方案 (Plan) -> 代码 (Implementation) -> 验证 (Verification)` 的结构。
2. **变更解释**: 对于每一处关键修改，必须在代码注释或回复中解释 **Reasoning (理由)**。
3. **Diff 友好**: 不要重新格式化未修改的代码片段，以免产生无意义的 Diff 干扰 Review。

# ai-jue Internal Governance

本文件定义 `ai-jue` 仓库内部协作约束，用于 `jue-preset-internal` 自举场景。

## 1. 目标

- 保障仓库持续迭代时的语义一致性（README/docs/spec/实现一致）。
- 保障变更质量（可验证、可回滚、可审查）。
- 保障最小知识原则（不引入额外用户概念）。

## 2. 执行原则

1. 先文档后实现：先修正文档语义，再改代码。
2. 架构优先：复杂变更先给设计边界与验证策略。
3. 小步提交：每次变更保持最小范围，确保可独立验证。
4. 向后兼容：默认保持兼容；破坏性变更必须显式标注。
5. 失败显式化：不允许静默降级或吞错。

## 3. 仓库治理检查清单

- 是否对齐当前 `TODO.md` 优先级与范围边界。
- 是否对齐 `_drafts/ai-jue.md` 的能力映射原则。
- 是否对齐 `README.md` 与 `packages/docs` 的规范叙述。
- 是否补齐相应测试/脚本验证与回归说明。

## 4. 预设边界

- `jue-preset-base`：提供通用用户场景能力。
- `jue-preset-internal`：仅提供本仓治理、自举与扩展开发约束。
- internal 不承载通用业务命令，不替代 base。

## 5. 适配与资产约定

- 规范目录：`AGENTS.md`、`commands/`、`rules/`、`hooks/`、`tools/`（按需启用）。
- `commands` 元数据统一写入 `prompt.md` YAML frontmatter。
- 不使用历史冗余概念与中间字段。

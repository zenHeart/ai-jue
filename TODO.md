# ai-jue 项目实施路线图 (TODO)

本文档基于“MVP 优先、配置自举、生态飞轮”的核心理念，对 `ai-jue` 项目的实施路径进行拆解。我们的目标是先用最小的代价验证核心闭环，然后快速利用工具自身实现经验沉淀，最终形成社区驱动的生态飞轮。

---

## Phase 0: MVP - 核心闭环与自我“自举”

**目标：** 用最少的代码验证“通过预设生成AI配置文件”这一核心流程，并立即应用到 `ai-jue` 项目自身的开发中。

- [ ] **基础 CLI 框架搭建**
    - [ ] 使用 `commander.js` 或 `yargs` 搭建基础命令，至少包含 `apply` 命令。
- [ ] **配置文件读取**
    - [ ] 使用 `cosmiconfig` 实现对配置文件的读取，**必须支持优先级顺序**：首先查找 `ai.config.js`(及相关变体)，若未找到，再查找 `jue.config.js` 作为备选。
- [ ] **最简预设加载**
    - [ ] 实现从 `node_modules` 中加载单个指定预设包的核心逻辑。
- [ ] **模板渲染与文件生成**
    - [ ] 实现 `apply` 命令，根据预设内容，使用模板引擎（如 Handlebars）生成目标文件。
    - [ ] **[关键要求]** 文件生成必须遵循“智能共存”策略：对 `.md` 文件采用“区块管理”，对 `.json` 文件采用“深度合并”，确保不粗暴覆盖用户修改。
- [ ] **创建第一个核心预设 `jue-preset-base`**
    - [ ] 创建该预设包，包含开发 `ai-jue` 项目自身所需的基础 prompts 和 skills。
- [ ] **[里程碑] 完成项目自我“自举”**
    - [ ] 在 `ai-jue` 项目中安装并配置 `jue-preset-base`。
    - [ ] 运行 `npx jue apply`，成功生成项目自身的 `GEMINI.md` / `CLAUDE.md` 等指导文件。
    - **至此，我们开始用自己的工具来沉淀开发自己的经验，第一个飞轮开始转动。**

---

## Phase 1: 可用性增强 & 本地资产管理

**目标：** 让工具对单个用户足够好用，打通“使用预设”与“沉淀本地经验”的链路。

- [ ] **实现 `.ai` 目录作为本地资产工作区（并兼容 `.jue` 作为备选）**
    - [ ] 约定 `.ai` / `.jue` 目录结构，如 `.ai/skills`, `.ai/prompts`。
    - [ ] `apply` 命令需要能识别并加载此目录下的本地资产，并优先使用 `.ai`。
- [ ] **完善配置继承与组合能力**
    - [ ] 在 `ai.config.js` 中实现 `extends` 字段，用于合并本地 `.ai` 目录的资产。
    - [ ] 支持 `presets: ['base', 'react']` 数组，实现多个预设的组合。
    - [ ] **[关键要求]** 加载和合并资产时，必须支持“环境特定性”，即实现对专属子目录（如 `claude/`）的识别与“覆写/回退”逻辑。
- [ ] **实现 `init` 交互式命令**
    - [ ] 运行 `npx jue init` 可以引导用户创建 `ai.config.js` 基础配置和 `.ai` 目录结构。
- [ ] **支持多语言 (i18n) 资产**
    - [ ] 在 `ai.config.js` 中增加顶层的 `language` 选项。
    - [ ] 在资产加载逻辑中，增加对语言子目录（如 `zh-CN/`）的支持，实现“覆写/回退”机制。
- [ ] **完善辅助命令**
    - [ ] 实现 `npx jue list`，可以列出当前已加载的预设和资产。
    - [ ] 实现 `npx jue check` 命令，用于检测预设更新并提示用户。

## Phase 2: 生态构建 - 预设打包与分享

**目标：** 打通“沉淀”到“分享”的路径，驱动社区生态飞轮。

- [ ] **[核心功能] 设计并实现 `create-preset` 命令**
    - [ ] `npx jue create-preset <preset-name>`
    - [ ] 命令能自动读取 `.ai` 目录下的本地资产。
    - [ ] 自动生成一个结构完整的、可直接发布的 npm 包（包含 `package.json`、入口文件、资产文件等）。
- [ ] **完善文档**
    - [ ] 撰写“如何创建和分享你的第一个预设”的详细指南。
- [ ] **建立预设展示渠道**
    - [ ] 在 `README.md` 或官方文档中开辟区域，用于展示和推荐优秀的社区预设。

---

## Phase 3: 架构进阶 & 工具链集成

**目标：** 提供更高级的功能，并融入更广泛的开发工具链中，提升项目的专业性和影响力。

- [ ] **[高级功能] 实现软链接模式**
    - [ ] 为 `apply` 命令增加 `--symlink` 选项，为高级用户提供更整洁的根目录管理方式。
    - [ ] 充分调研并解决跨平台兼容性问题。
- [ ] **[效率工具] 开发 IDE 扩展 (VS Code)**
    - [ ] 提供对 `ai.config.js` 的语法高亮和智能提示。
    - [ ] 提供一键运行 `apply`, `init` 等命令的功能。
- [ ] **[生态集成] 探索与脚手架工具的集成**
    - [ ] 研究如何在 `create-vite`, `create-next-app` 等工具的执行流程中，加入 `npx jue init` 步骤。
- [ ] **[健壮性] 实现 `validate` 命令**
    - [ ] `npx jue validate` 用于检查 `ai.config.js` 配置的正确性及预设的结构是否符合规范。

## Phase 4 / 远期战略思考

**目标：** 解决项目在走向成熟和大规模应用时，必然会遇到的系统性、生态性问题，确保其长期生命力。

- [ ] **生态系统与治理**
    - [ ] **预设发现与信任：** 建立预设的官方认证、质量评分、社区排行榜等策展机制。
    - [ ] **命名空间与废弃管理：** 推广并固化 `@scope` 命名约定；建立预设的废弃与社区领养流程。

- [ ] **开发者工作流的无缝集成**
    - [ ] **热重载体验：** 实现 `npx jue apply --watch` 模式，实时同步配置变更。
    - [ ] **源映射与调试：** 探索在生成文件中注入“源映射”注释，并配合 IDE 插件实现来源追溯，彻底改变配置调试体验。

- [ ] **跨工具抽象的挑战**
    - [ ] **编译器架构：** 将 `ai-jue` 核心重构为“抽象能力描述语言 + 多目标后端编译器”的模式，并为此引入真正的“生成器插件”(`generator-plugin`)机制。
    - [ ] **逃生通道：** 在 `ai.config.js` 中提供特定于工具的“直通”配置能力，以支持抽象层无法覆盖的独有功能。

- [ ] **配置语言的表达力**
    - [ ] **高级操作符：** 探索支持在配置中对继承的资产进行“禁用”或“修改”(`patch`) 的能力，以应对更复杂的定制化需求。
